# Spring Framework 탄생 배경과 프레임워크 vs 라이브러리

## 1. Spring Framework의 탄생 배경

Spring Framework는 초기 Java 엔터프라이즈 개발의 여러 문제점들을 해결하기 위해 탄생했습니다.

### 주요 해결 과제

- **EJB의 복잡성 개선**: 당시 표준이었던 EJB(Enterprise JavaBeans) 기술이 지나치게 복잡하고 무거웠습니다. Spring은 더 쉽고 가벼운 개발 방식을 제공하고자 했습니다.

- **강한 결합(Tight Coupling) 문제**: 초기 자바 엔터프라이즈 애플리케이션은 컴포넌트 간 강하게 결합되어 있어 테스트와 유지보수가 어려웠습니다. Spring은 느슨한 결합(Loose Coupling)과 높은 응집력(High Cohesion)을 달성하기 위한 방법론을 제공했습니다.

- **복잡한 트랜잭션 관리**: 개발자가 트랜잭션을 수동으로 처리해야 했기 때문에 오류가 발생하기 쉽고 코드가 복잡했습니다. Spring은 선언적 트랜잭션 관리를 통해 이를 단순화했습니다.

## 2. 프레임워크 vs 라이브러리

### 제어 흐름의 주체 차이

- **라이브러리**: 개발자가 제어 흐름의 주체입니다. 개발자가 필요한 시점에 라이브러리의 기능을 호출하여 사용합니다.

- **프레임워크**: 프레임워크 자체가 제어 흐름을 주도합니다(IoC: Inversion of Control). 프레임워크가 애플리케이션의 구조를 정의하고, 개발자는 그 구조 안에서 코드를 작성합니다.

### 사용 방식의 차이

#### 라이브러리
- 개발자가 필요에 따라 선택적으로 사용할 수 있으며, 애플리케이션 구조에 영향을 주지 않습니다.
- **예시**: Apache Commons, Gson 같은 일반 Java 라이브러리

#### 프레임워크
- 애플리케이션의 전체 구조와 흐름을 결정하며, 개발자는 프레임워크가 정한 규칙을 따라야 합니다.
- **예시**: Spring Framework는 DI 컨테이너를 통해 객체 생성과 생명주기를 관리하고, 개발자는 Spring이 정한 방식(어노테이션, 설정 등)에 따라 개발합니다.

## 3. Spring Framework vs 일반 Java 라이브러리 비교 예시

**Java 라이브러리 특징:**
- 개발자가 언제, 어떻게 사용할지 완전히 제어
- 애플리케이션 구조에 영향을 주지 않음
- 필요한 부분에서만 선택적으로 사용


**Spring Framework 특징:**
- Spring이 애플리케이션 시작 시 객체를 생성하고 관리 (IoC Container)
- Spring이 HTTP 요청을 받아 적절한 메서드를 호출 (제어 흐름 주도)
- 개발자는 `@RestController`, `@Autowired`, `@GetMapping` 등 Spring이 정한 규칙을 따라야 함
- 애플리케이션의 전체 구조가 Spring의 방식을 따름
