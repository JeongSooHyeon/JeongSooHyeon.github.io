# Git Merge vs Git Rebase 정리

## 1. 개요
`git merge`와 `git rebase`는 모두 브랜치를 합칠 때 사용하는 명령어지만  
**히스토리를 다루는 방식과 사용 목적이 다르다.**

- Merge: 기록 보존에 유리
- Rebase: 히스토리 파악에 유리  
👉 정답은 없으며, **팀 컨벤션에 맞추는 것이 가장 중요하다.**

---

## 2. Git Merge

### 🔹 개념
- 두 개의 브랜치를 병합하면서 **새로운 merge commit을 생성**
- 기존 커밋 히스토리를 **그대로 유지**

### 🔹 히스토리 특징
- 브랜치가 분기되고 병합된 과정이 모두 남음
- 히스토리가 트리 구조로 남아 **복잡해질 수 있음**


### 🔹 장점
- 커밋 해시가 변경되지 않음
- 어떤 브랜치에서 어떤 작업을 했는지 명확히 추적 가능
- 협업 시 안전함

### 🔹 단점
- merge commit이 많아지면 히스토리가 복잡해질 수 있음

### 🔹 사용하기 적절한 상황
- 공용 브랜치(`main`, `master`)에 병합할 때
- 협업 프로젝트
- 작업 흐름과 기록을 명확히 남겨야 하는 경우

---

## 3. Git Rebase

### 🔹 개념
- 현재 브랜치의 **base를 대상 브랜치의 최신 커밋으로 변경**
- 커밋들을 다시 쌓아 **히스토리를 일자로 정리**

### 🔹 히스토리 특징
- 선형(Linear) 히스토리
- 깔끔하고 가독성이 좋음


### 🔹 장점
- 히스토리가 깔끔해서 파악이 쉬움
- merge commit 없이 커밋이 정리됨
- 모든 커밋이 메인 브랜치에 정돈되어 들어감

### 🔹 단점
- **커밋 해시가 변경됨**
  - 부모 커밋이 바뀌어 해시를 다시 계산
- 협업 시 히스토리 추적이 어려울 수 있음
- 이미 공유된 브랜치에서 사용하면 위험

### 🔹 사용하기 적절한 상황
- 개인 작업 브랜치 정리
- PR 전에 커밋을 깔끔하게 정리하고 싶을 때
- 모든 커밋을 master/main에 정돈되게 넣고 싶을 때
- 병합 커밋 없이 선형 히스토리를 유지하고 싶을 때

---

## 4. Merge vs Rebase 비교

![rebase vs merge](/assets/images/rebase_merge.png)


| 구분 | Merge | Rebase |
|----|----|----|
| 히스토리 | 트리 구조 (기록 유지) | 선형 구조 (깔끔) |
| merge commit | 생성됨 | 생성 안 됨 |
| 커밋 해시 | 유지됨 | 변경됨 |
| 협업 안정성 | 높음 | 낮음 |
| 주 사용처 | 공용 브랜치 병합 | 개인 브랜치 정리 |

---

## 5. 언제 무엇을 쓰는 게 좋은가?

### ✅ Git Merge를 사용해야 할 때
- 협업 중인 브랜치
- 메인 브랜치에 기능 브랜치 병합
- 히스토리 기록이 중요한 프로젝트

### ✅ Git Rebase를 사용해야 할 때
- 개인 작업 브랜치
- PR 전 커밋 정리
- 히스토리를 깔끔하게 유지하고 싶은 경우

---

## 6. 결론
- **Merge는 기록 보존에 유리**
- **Rebase는 히스토리 파악에 유리**
- 정답은 없으며, **팀 내 컨벤션과 협업 방식에 맞추는 것이 가장 중요하다**
