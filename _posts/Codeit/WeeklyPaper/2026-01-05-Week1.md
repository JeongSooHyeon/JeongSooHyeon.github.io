git rebase와 git merge의 차이점을 설명하고, 각각 어떤 상황에서 사용하는 것이 적절한지 설명해주세요.   
- 브랜치를 합칠 때 commit을 일자로 피면서 합칠 때
- 현재 branch의 base를 변경, 대상 브랜치의 최신 커밋이 현재 브랜치의 베이스가 된다
- 단점 : 해시값 변경됨(부모 commit이 바껴서 해시값 다시 계산됨, history 파악할 때 해시값 기록해놓고 파악하면 헷갈려짐)
	- 협업 시 히스토리 파악이 어려울 수 있다. 각각의 commit이 어떤 branch에서 나와서 어떻게 합쳐졌는지 안 보이게 됨
- 모든 커밋 하나하나가 master에 정리되어 들어가야 한다면 rebase 써야 함
- merge는 두 개의 브랜치 병합하여 새로운 커밋 생성, 주로 기능 개발이 완료된 브랜치를 메인 브랜치에 병합, 히스토리 유지
- rebase 히스토리를 일자로 유지하여 가독성 높아짐
- merge : 히스토리를 그대로 유지해야하는 프로젝트, 각 작업의 진행 과정을 명확히 추적해야 하는 경우
- rebase : 병합 커밋 없이 커밋 히스토리를 깔끔하고 선형적으로 유지하고 싶을 때, 충돌 발생할 수 있음, 롤백이 예상되는 경우


객체지향 프로그래밍에서 '단일 책임 원칙(SRP)'과 '개방-폐쇄 원칙(OCP)'에 대해 설명하고, 각각의 원칙을 적용한 코드 예시를 들어주세요.
- srp : 한 클래스는 하나의 책임만 가져야 한다. 어떤 변화에 의해 클래스를 변경해야하는 이유는 오직 하나 뿐이어야한다.  
- ocp : 소프트웨어 요소는 확장에는 열려 있으나 변경에는 닫혀 있어야 한다.
객체지향의 추상화와 다형성을 활용해야 한다.
