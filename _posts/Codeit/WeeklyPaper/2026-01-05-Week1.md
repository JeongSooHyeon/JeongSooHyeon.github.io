# Git Merge vs Git Rebase 정리

## 1. 개요
`git merge`와 `git rebase`는 모두 브랜치를 합칠 때 사용하는 명령어지만,  
**히스토리를 다루는 방식과 사용 목적이 다르다.**

- **Merge**: 기록 보존에 유리
- **Rebase**: 히스토리 파악에 유리  

👉 정답은 없으며, **팀 컨벤션에 맞추는 것이 가장 중요하다.**

---

## 2. Git Merge

### 🔹 개념
- 두 개의 브랜치를 병합하면서 **새로운 merge commit을 생성**
- 기존 커밋 히스토리를 **그대로 유지**

### 🔹 히스토리 특징
- 브랜치가 분기되고 병합된 과정이 모두 남음
- 히스토리가 트리 구조로 남아 **복잡해질 수 있음**

### 🔹 장점
- 커밋 해시가 변경되지 않음
- 어떤 브랜치에서 어떤 작업을 했는지 추적 가능
- 협업 시 안전함

### 🔹 단점
- merge commit이 많아지면 히스토리가 복잡해질 수 있음

### 🔹 사용하기 적절한 상황
- 공용 브랜치(`main`, `master`)에 병합할 때
- 협업 프로젝트
- 작업 흐름과 기록을 명확히 남겨야 하는 경우

---

## 3. Git Rebase

### 🔹 개념
- 현재 브랜치의 **base를 대상 브랜치의 최신 커밋으로 변경**
- 기존 커밋들을 다시 쌓아 **히스토리를 일자로 정리**

### 🔹 히스토리 특징
- 선형(Linear) 히스토리
- 깔끔하고 가독성이 좋음

### 🔹 장점
- 히스토리가 깔끔하여 파악이 쉬움
- merge commit 없이 커밋이 정리됨

### 🔹 단점
- **커밋 해시가 변경됨**
- 협업 중인 브랜치에서 사용 시 위험
- 히스토리 추적이 어려워질 수 있음

### 🔹 사용하기 적절한 상황
- 개인 작업 브랜치 정리
- PR 전 커밋을 깔끔하게 정리하고 싶을 때
- 선형 히스토리를 유지하고 싶을 때

---

## 4. Merge vs Rebase 비교

![rebase vs merge](/assets/images/rebase_merge.png)

| 구분 | Merge | Rebase |
|---|---|---|
| 히스토리 | 트리 구조 | 선형 구조 |
| merge commit | 생성됨 | 생성 안 됨 |
| 커밋 해시 | 유지됨 | 변경됨 |
| 협업 안정성 | 높음 | 낮음 |
| 주 사용처 | 공용 브랜치 | 개인 브랜치 |

---

## 5. 결론
- **Merge는 기록 보존에 유리**
- **Rebase는 히스토리 정리에 유리**
- 팀 컨벤션과 협업 방식에 맞춰 선택하는 것이 가장 중요하다

---

# Git Fetch vs Git Pull 정리

## 1. 개요
`git fetch`와 `git pull`은 모두 **원격 저장소의 변경 사항을 확인하거나 가져올 때** 사용하지만,  
**로컬 브랜치에 반영되는 방식에서 큰 차이**가 있다.

---

## 2. Git Fetch

### 🔹 개념
- 원격 저장소의 **최신 메타데이터만 가져오는 명령어**
- 로컬 브랜치의 코드는 **전혀 변경되지 않음**

### 🔹 특징
- 원격 브랜치의 최신 커밋 정보만 업데이트
- 실제 병합은 발생하지 않음
- 변경 내역을 안전하게 사전 확인 가능

### 🔹 사용하기 적절한 상황
- 팀원들의 변경 내역을 먼저 확인하고 싶을 때
- 자동 merge가 불안할 때
- 여러 브랜치 상황을 파악하고 싶을 때
- 충돌 가능성을 최소화하고 싶을 때

---

## 3. Git Pull

### 🔹 개념
- **`git fetch` + `git merge`를 한 번에 수행**
- 원격 저장소의 변경 사항을 가져와 **즉시 현재 브랜치에 반영**

### 🔹 특징
- 빠르게 최신 상태로 작업 가능
- 내 작업과 바로 섞이기 때문에 충돌 가능성 존재
- 변경 내역을 검토할 여유가 없음

### 🔹 사용 시 주의점
- 협업 중일 경우 예상치 못한 충돌 발생 가능
- 변경 사항을 바로 반영해도 괜찮은 상황에서만 사용 권장

---

## 4. 안전한 협업 흐름

1. `git fetch`로 원격 저장소 상황 확인
2. 변경 내역 검토
3. `git merge` 또는 `git rebase` 여부 결정

---

## 5. 최종 정리

- `git fetch`는 **확인용**
- `git pull`은 **즉시 반영**
- 협업 환경에서는  
  **fetch → 판단 → merge/rebase** 흐름이  
  가장 안전하고 충돌을 줄일 수 있다
